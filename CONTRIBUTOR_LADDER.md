## ğŸ¯ Philosophy of Growth

### Core Principles

1. **Growth Through Contribution**
   - Advancement based on demonstrated skill and judgment, not tenure
   - Different paths to leadership (code, docs, design, community)
   - Contribution > Job Title (volunteers can become maintainers)

2. **Servant Leadership**
   - Higher levels = more service, not more power
   - Leadership is stewardship of project and community
   - Authority comes from trust earned, not title granted

3. **Sacred Reciprocity**
   - Project benefits from contributions
   - Contributors grow through participation
   - Community knowledge flows in all directions

4. **Conscious Succession**
   - Develop future leaders continuously
   - Leadership is distributed, not concentrated
   - Emeritus status honors those stepping back

5. **Multiple Paths**
   - Not everyone wants to be a maintainer (and that's perfect!)
   - Value contributors at every level
   - Lateral movement between roles

## ğŸ‘€ Level 1: Observer

### Who Are You?

- **New to the project** - Learning what we do and how
- **Exploring fit** - Deciding if this aligns with your interests
- **Building context** - Understanding codebase, community, values

### What You Can Do

**Learn:**
- Read documentation ([GETTING_STARTED.md](./GETTING_STARTED.md))
- Browse codebase and past discussions
- Join community channels (Discord, Discussions)
- Attend office hours and community calls

**Participate:**
- Ask questions in Discussions
- Comment on issues and PRs (constructive feedback welcome)
- Vote/react to feature proposals
- Lurk and learn (no pressure to speak up)

**Small Contributions:**
- Typo fixes in documentation
- Add yourself to [CONTRIBUTORS.md](./CONTRIBUTORS.md)
- Report bugs (with details)

### Responsibilities

- Follow [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md)
- Be respectful and curious
- Assume good intent

### How to Advance

**Ready for Contributor when you:**
- [ ] Understand project values (ERC, consciousness-first)
- [ ] Know where things are (docs, code structure, communication channels)
- [ ] Made first small contribution (even tiny PR merged)
- [ ] Feel excited to contribute more

**No application needed** - Make your first non-trivial contribution and you're a Contributor!

## ğŸŒ± Level 2: Contributor

### Who Are You?

- **Active participant** - Making regular contributions
- **Learning by doing** - Growing skills through real work
- **Building reputation** - Demonstrating judgment and values alignment

### What You Can Do

### Responsibilities

- **Communication** - Respond to review feedback promptly

- **Community** - Help newcomers, share knowledge

### Support You Get

- **Mentorship** - Pair with experienced contributors
- **Recognition** - Contributions celebrated publicly
- **Learning resources** - Access to learning materials and workshops

### How to Advance

**Ready for Regular Contributor when you:**
- [ ] **Consistent contributions** - 5+ merged PRs over 2+ months
- [ ] **Quality work** - Code is well-tested, documented, reviewed positively
- [ ] **Good judgment** - Understanding when to ask vs. proceed
- [ ] **Community alignment** - Contributions reflect ERC values
- [ ] **Helping others** - Answering questions, welcoming newcomers

**No formal application** - Maintainers will notice and may invite you to Regular Contributor conversations.

### What You Can Do

- **Strategic decisions** - Vote on major project directions
- **Policy creation** - Help create and update governance policies
- **Budget input** - Influence how funds are allocated
- **Roadmap planning** - Shape quarterly and annual priorities

### Responsibilities

**Core Responsibilities (8-15 hours/week typical):**

**Code Quality:**
- [ ] Review PRs within expected timeframes (see [CODE_REVIEW.md](./CODE_REVIEW.md))
- [ ] Maintain high quality bar while being welcoming
- [ ] Ensure security and performance standards met
- [ ] Keep technical debt manageable

- [ ] Welcome and mentor new contributors
- [ ] Respond to community questions and concerns
- [ ] Foster inclusive, values-aligned culture
- [ ] Address Code of Conduct issues promptly

**Project Health:**
- [ ] Monitor issues and triage regularly
- [ ] Keep documentation current
- [ ] Participate in planning and retrospectives
- [ ] Maintain awareness of ecosystem changes

**Communication:**
- [ ] Attend monthly maintainer sync (or send update)
- [ ] Document important decisions in ADRs
- [ ] Be available for emergency response
- [ ] Communicate context to other maintainers

### Support You Get

- **Compensation** (when funds allow):
  - Priority for paid development work
  - Conference sponsorship and speaking opportunities
  - Stipend or hourly rates for maintenance work
- **Community support**:
  - Backup on-call rotation
  - Collaboration with other maintainers
  - Respect and gratitude from community
- **Professional growth**:
  - Leadership development
  - Speaking and writing opportunities
  - Network with other open source leaders

### Onboarding Checklist

When you become a Reviewer/Maintainer:

**Week 1: Access & Setup**
- [ ] GitHub org admin access granted
- [ ] Added to maintainer Discord channels
- [ ] NPM/PyPI publish access (if applicable)
- [ ] Infrastructure access (CI/CD, monitoring)
- [ ] Read all governance docs (ADR, GOVERNANCE, etc.)
- [ ] 1:1 with existing maintainer (welcome, questions)

**Week 2-4: Shadow & Learn**
- [ ] Shadow PR reviews (comment, don't merge yet)
- [ ] Attend maintainer sync meeting
- [ ] Participate in issue triage
- [ ] Review release process documentation

**Month 2: Full Responsibility**
- [ ] First independent PR merge
- [ ] First release cut (with guidance)
- [ ] First ADR authored or co-authored
- [ ] Added to project maintainers list publicly

**Ongoing:**
- [ ] Regular 1:1s with mentor maintainer (first 3 months)
- [ ] Attend monthly maintainer syncs
- [ ] Participate in quarterly retrospectives

### How to Advance

**Ready for Core Team when you:**
- [ ] **Long-term commitment** - 18+ months as maintainer
- [ ] **Strategic vision** - Shape project direction, not just implement
- [ ] **Ecosystem leadership** - Known beyond project, represent org externally
- [ ] **Mentorship excellence** - Grown multiple contributors to maintainer level
- [ ] **Organizational thinking** - Consider cross-project implications
- [ ] **Conscious leadership** - Embody ERC values in all interactions

**Invitation process:**
1. Core Team discusses need and fit
2. Invitation extended
3. Acceptance and onboarding

## ğŸ“ Emeritus Status

### What Is Emeritus?

**For those who have served and are stepping back:**
- Life circumstances change (job, family, health, other priorities)
- Burnout prevention (better to step back than burn out)
- Natural transitions (project matured, personal growth direction shifted)

**Emeritus = Honored Alumni**

### Emeritus Privileges

- **Title and recognition** - "Emeritus Maintainer" or "Emeritus Core Team"
- **Access retained** - Can still comment, review, contribute (not merge/vote)
- **Advisory role** - Input sought on major decisions
- **Return pathway** - Can return to active status when ready
- **Community membership** - Still part of the family

### How to Transition

**From Active to Emeritus:**
1. Communicate intention to Core Team (private or public)
2. 1-3 month transition period (hand off responsibilities)
3. Announcement celebrating contributions
4. Emeritus status granted

**From Emeritus to Active:**
- Express interest in returning
- Refresh on changes since departure
- Reactivate access and responsibilities
- Welcome back!

## ğŸ’š Conscious Leadership Principles

### Leadership as Service

**At every level, leadership means:**

1. **Empowering Others**
   - Make it easier for others to contribute
   - Share knowledge generously
   - Create space for others to lead

2. **Stewardship Over Ownership**
   - You're caring for project temporarily
   - Success = project thrives without you
   - Develop your successors

3. **Transparency**
   - Decisions made openly
   - Reasoning documented
   - No backroom deals

4. **Humility**
   - Admit mistakes quickly
   - Learn from everyone
   - No egos, no heroes

5. **Care for Whole System**
   - Contributors, users, environment, future
   - Not just code quality, but community health
   - Long-term thinking

### Anti-Patterns to Avoid

âŒ **Gatekeeping**: Using authority to block instead of guide
âŒ **Hero Culture**: Doing everything yourself instead of delegating
âŒ **Burnout**: Saying yes to everything until you collapse
âŒ **Scope Creep**: Growing your role beyond healthy bounds
âŒ **Hoarding Knowledge**: Keeping critical knowledge in your head

âœ… **Instead:**
- Guide contributors to solutions
- Delegate and trust
- Set boundaries, step back when needed
- Keep role focused
- Document everything
